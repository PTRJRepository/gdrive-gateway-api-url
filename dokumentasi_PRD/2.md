Tentu, ini adalah **Product Requirements Document (PRD)** versi Bahasa Indonesia untuk pengembangan sistem **API Storage Gateway** dengan backend Google Drive.

Dokumen ini dirancang agar tim engineering memiliki panduan teknis yang jelas untuk membangun lapisan abstraksi di atas Google Drive API.

---

# ## PRD: API Storage Gateway (Google Drive Backend)

**Versi:** 1.0

**Status:** Draft / Perancangan

**Pemilik Produk:** AI Collaborator / [Nama Anda]

---

### ### 1. Analisis Masalah & Tujuan

Banyak aplikasi membutuhkan penyimpanan cloud, namun berinteraksi langsung dengan Google Drive API v3 seringkali rumit karena masalah OAuth2, limitasi kecepatan (rate limiting), dan manajemen File ID yang tidak intuitif.

**Tujuan Utama:**

* Menyediakan API yang sederhana (S3-like) untuk tim internal.
* Menyembunyikan kompleksitas autentikasi Google (Service Accounts).
* Mengelola streaming file agar tidak membebani memori server (Gateway).

---

### ### 2. Arsitektur Sistem (API-Based)

Sistem ini bertindak sebagai **Proxy** antara Client (Frontend/Microservice lain) dan Google Drive.

---

### ### 3. Kebutuhan Fungsional (Functional Requirements)

| Fitur | Deskripsi |
| --- | --- |
| **FR1: Resumable Upload** | Mendukung upload file besar (>10MB) dengan metode chunking agar tidak putus di tengah jalan. |
| **FR2: Download Proxy** | Menyediakan endpoint download yang mengalirkan (streaming) data dari G-Drive ke client tanpa membocorkan URL asli G-Drive. |
| **FR3: Folder Mapping** | Kemampuan untuk menentukan folder tujuan di G-Drive secara dinamis berdasarkan parameter API. |
| **FR4: Auth Gateway** | Autentikasi internal menggunakan JWT atau API Key sebelum permintaan diteruskan ke Google. |
| **FR5: Metadata DB** | Menyimpan pemetaan antara `Internal_ID` (milik sistem kita) dengan `GDrive_File_ID`. |

---

### ### 4. Alur Kerja Utama (User Stories & Logic)

#### **4.1. Alur Upload (Client → Gateway → G-Drive)**

1. **Inisiasi:** Client mengirim metadata file ke `/v1/upload/init`.
2. **Session:** Gateway meminta `Upload Session URI` dari Google dan memberikan ID unik ke Client.
3. **Streaming:** Client mengirim data biner ke Gateway, yang langsung di-pipe (dialirkan) ke Google Cloud tanpa disimpan permanen di disk Gateway.

#### **4.2. Alur Download**

1. Client memanggil `GET /v1/download/{file_id}`.
2. Gateway mencari `GDrive_File_ID` di database metadata.
3. Gateway melakukan stream request ke Google dan mem-forward bit data tersebut ke response client secara real-time.

---

### ### 5. Spesifikasi API (Draft)

* **POST `/v1/files/upload**`
* *Body:* `multipart/form-data` (File, Target Folder)
* *Response:* `201 Created` dengan `file_id` internal.


* **GET `/v1/files/download/{id}**`
* *Response:* Stream file biner.


* **DELETE `/v1/files/{id}**`
* *Response:* `204 No Content` (menghapus file di G-Drive dan record di DB).



---

### ### 6. Kebutuhan Non-Fungsional (Non-Functional Requirements)

* **Keamanan:** Semua komunikasi wajib menggunakan HTTPS. Token Google Service Account harus disimpan di Secret Manager (Vault/GCP Secret).
* **Ketersediaan (Availability):** Stateless design agar gateway bisa di-scale secara horizontal (Kubernetes/Docker).
* **Kinerja:** Response time untuk inisiasi session harus di bawah 200ms.
* **Error Handling:** Implementasi **Exponential Backoff** jika terkena limitasi API Google (Error 423/429).

---

### ### 7. Metrik Keberhasilan

1. **Latency:** Rata-rata waktu tunggu client saat memulai upload.
2. **Error Rate:** Persentase kegagalan upload karena gangguan jaringan atau limitasi API.
3. **Storage Efficiency:** Konsumsi memori RAM server Gateway tetap rendah (dibawah 512MB) meskipun menangani file berukuran GB (karena sistem streaming).
