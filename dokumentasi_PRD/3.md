# DOKUMEN PRD & RANCANG BANGUN SISTEM
**Nama Produk:** Enterprise File Gateway API
**Tipe Sistem:** RESTful API Backend / Microservice 
**Teknologi Utama:** Node.js (Express) atau Python (FastAPI), Google Drive API, Google Cloud IAM (Service Accounts)

## 1. Product Requirements Document (PRD)

### 1.1. Ringkasan Eksekutif
Sistem ini adalah gerbang file (file gateway) tingkat enterprise yang menyediakan antarmuka terpusat antara logika aplikasi lokal dan repositori penyimpanan cloud (Google Drive). Sistem ini sepenuhnya beroperasi di sisi server menggunakan *Service Account* sehingga tidak memerlukan intervensi login dari pengguna (headless authentication).

### 1.2. Tujuan Sistem
*   Membangun REST API yang *scalable* untuk proses unggah (ingestion), penemuan (discovery), pengambilan (retrieval), dan modifikasi file.
*   Memastikan resiliensi layanan dari kegagalan jaringan atau limitasi kuota API melalui implementasi asinkron dan algoritma *exponential backoff*.
*   Mendukung pengelolaan *pooled storage* Google Workspace secara efisien agar kapasitas data (hingga hitungan Terabyte) tidak mengganggu operasional sistem.

### 1.3. Kebutuhan Fungsional (Fitur Utama API)
Sistem akan menyediakan endpoint REST API (URLs) untuk kebutuhan berikut:
1.  **File Ingestion (Upload):**
    *   Sistem mendukung *Simple Upload* (file di bawah 5MB) dan *Multipart Upload* (metadata & file sekaligus).
    *   Sistem **wajib** mendukung *Resumable Upload* untuk file berukuran besar (mendukung hingga 5 TB) guna mencegah kegagalan jika koneksi terputus di tengah jalan.
2.  **File Retrieval & Search (Baca/Cari):**
    *   Dapat mencari file menggunakan kueri khusus dan menerapkan logika paginasi (`nextPageToken`) untuk memproses jutaan item tanpa *time-out*.
    *   Dapat mengunduh file biner (`alt=media`) dan mengekspor dokumen Google Workspace menjadi PDF/Office.
3.  **Modifikasi & Manajemen Data:**
    *   Mendukung pembaruan metadata (nama, visibilitas) menggunakan semantik *Patch* sehingga tidak merusak isi file.
    *   Dapat menyimpan ID file Google Drive dan *webviewlink* ke dalam database internal (seperti MongoDB/PostgreSQL) sebagai referensi bagi pengguna akhir.

### 1.4. Manajemen Autentikasi dan Keamanan
*   **Otorisasi API Internal:** API Gateway lokal diamankan menggunakan JWT (JSON Web Tokens) akses yang valid pada *header request* pengguna (berlaku 30 menit, dan dapat di-*refresh*).
*   **Otorisasi ke Google Drive:** Menghubungkan *Service Account* ke Google API dengan menggunakan kredensial berupa file JSON. Akses Drive diterapkan menggunakan prinsip hak istimewa terkecil (Least Privilege) melalui model *Explicit Folder Sharing*.

## 2. Dokumen Rancang Bangun Sistem (System Architecture)

### 2.1. Arsitektur Aplikasi
Sistem akan diimplementasikan sebagai *Microservice* berbasis *asynchronous non-blocking I/O* untuk memaksimalkan performa transfer file bersamaan. Pendekatan kerangka kerja dapat berupa:
*   **Opsi A (Python / FastAPI):** Memanfaatkan keunggulan *Starlette* untuk konkurensi (event loop) yang tinggi dan validasi struktur data bawaan melalui *Pydantic*.
*   **Opsi B (Node.js / Express):** Memanfaatkan boilerplate produksi yang mencakup koneksi MongoDB (Mongoose), validasi input dengan Joi, dan manajemen proses via PM2.

### 2.2. Pola Desain Kode (Design Pattern)
Pola yang digunakan untuk penulisan API adalah pemisahan logika empat lapis (Layered Architecture):
1.  **Routes / Endpoints:** Titik masuk URL dari REST API tanpa logika bisnis (contoh: `POST /v1/files/upload`).
2.  **Controllers:** Mengorkestrasi alur kerja, memvalidasi input (dengan Joi/Pydantic), dan memanggil Service layer.
3.  **Services:** Memuat seluruh operasi inti bisnis, termasuk integrasi ke SDK Google Drive API.
4.  **Repositories:** Melakukan operasi akses database lokal (CRUD metadata ke MongoDB/SQL).

### 2.3. Penanganan Error & Reliabilitas (Error Handling)
Kestabilan dan keandalan sistem dibangun dengan prinsip berikut:
*   **Format Respons Error Terpusat:** Semua error pada API URL harus dikembalikan dalam format struktur JSON standar (mengandung field kode, status, deskripsi masalah, dan rekomendasi perbaikan). Error *stack trace* hanya dimunculkan dalam mode *development*.
*   **Exponential Backoff:** Setiap request ke Google API yang menerima *HTTP 403 Rate Limit Exceeded*, *429 Too Many Requests*, atau *500-level error*, wajib diulang otomatis (*retry*) dengan jeda waktu eksponensial (misalnya rumus $WaitTime = \min(2^{retry\_count} + \text{random\_ms}, 64)$) untuk mencegah beban berlebih.
*   **Session Persistence:** Khusus unggahan *resumable*, URI sesi yang dihasilkan oleh Google akan disimpan pada sistem *cache* terdistribusi (seperti Redis). Jika satu instansi server mati, unggahan dapat diteruskan oleh server lain (Load Balanced).

### 2.4. Manajemen Kuota dan Infrastruktur Storage
*   **Rotasi & Pengawasan Kunci:** Kunci JSON Service Account (*private_key*) wajib disimpan secara rahasia di dalam *Google Secret Manager* dan tidak boleh di-*commit* ke repositori (contoh: GitHub).
*   **Pemantauan Kapasitas (Quota-Aware):** API akan memonitor batas penyimpanan harian maksimum dari Google Workspace. Contohnya, batasan unggahan/salinan harian Google Drive adalah 750 GB per akun. Jika kuota hampir tercapai (misal peringatan pada kapasitas 80%-90%), sistem akan memicu *alert* log otomatis (menggunakan Cloud Logging) kepada administrator.
